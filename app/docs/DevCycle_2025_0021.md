# Development Cycle 2025-0021

**Status:** 🎫 Open
**Start Date:** 2025-12-07
**Target Completion:** TBD
**Focus:** Piece Metadata Enhancement and Pro Mode Configuration

## Overview

This development cycle focuses on enhancing piece/technique metadata with additional musical information fields (Key, Artist, and Notes) and adding user control over Pro mode functionality through a configuration toggle. These enhancements will improve the app's utility for musicians tracking their repertoire while giving users explicit control over premium features.

The following specifications apply across all phases implementing the Key, Artist, and Notes fields:

**Database Migration (Q4 Answer - Simple ALTER TABLE):**
- Current database version: 2 (from DevCycle 2025-0004)

- Migration strategy: Simple ALTER TABLE ADD COLUMN approach
- Migration SQL:
  ```sql
  ALTER TABLE pieces_techniques ADD COLUMN key TEXT;
  ALTER TABLE pieces_techniques ADD COLUMN artist TEXT;
  ALTER TABLE pieces_techniques ADD COLUMN notes TEXT;
  ```

- All three columns are nullable (NULL default for existing records)

**Key Field Specification (Q1 Answer):**
- **UI Component**: Dropdown/Spinner (not free-form text input)
- **Options**: Keys following the circle of fifths: C, G, D, A, F, B, Cb, F#, Gb, Db, C#, Ab, Eb, Bb, F
- **Format**: Both Major and Minor for each key (e.g., "C Major", "C Minor", "G Major", "G Minor")
- **Complete List**:
  - C Major, C Minor
  - G Major, G Minor
  - D Major, D Minor
  - A Major, A Minor
  - E Major, E Minor
  - B Major, B Minor
  - F# Major, F# Minor
  - Db Major, Db Minor (enharmonic with C#)
  - Ab Major, Ab Minor
  - Eb Major, Eb Minor
  - Bb Major, Bb Minor
  - F Major, F Minor
- **Optional**: Includes "None" or empty option for pieces without specified key

**Artist Field Specification (Q9 Answer - Free-form Text):**

- **Position in Display**: Before Key field (Q10 Answer)

- **Display**: Multiline support with proper text area

**Configuration Fragment Implementation (Q6 Answer - Reset to Non-Pro):**
```kotlin
// In ConfigurationFragment.kt
override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    super.onViewCreated(view, savedInstanceState)

    // Initialize Pro Mode toggle with current state
    val proUserManager = ProUserManager(requireContext())

    // Q6: Reset all users to non-Pro on first launch after this feature is added
    // The toggle will default to unchecked (false), and users must explicitly opt-in
    binding.proModeToggle.isChecked = proUserManager.isProUser()

    // Handle toggle changes
    binding.proModeToggle.setOnCheckedChangeListener { _, isChecked ->
        proUserManager.setProUser(isChecked)
        Toast.makeText(
            requireContext(),
            if (isChecked) "Pro Mode enabled" else "Pro Mode disabled",
            Toast.LENGTH_SHORT
        ).show()
    }
}
```
**Migration Note**: When this feature is deployed, all users (including those who may have had Pro mode enabled previously) will be reset to non-Pro mode and must explicitly enable it via the toggle (Q6 Answer).
